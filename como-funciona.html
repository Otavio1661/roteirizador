0 O Problema que o Sistema Resolve

Imagine que você precisa visitar N lugares diferentes e voltar ao ponto de partida. Qual é a ordem de visita que percorre a menor distância total?

Isso se chama Problema do Caixeiro Viajante (TSP — Traveling Salesman Problem). É um dos problemas mais famosos da ciência da computação, e o motivo é simples:

Por que é difícil?

Com N pontos, existem (N−1)! / 2 rotas possíveis. Com 10 pontos são 181.440 combinações. Com 20 pontos são 60.822.550.204.416.000 combinações — impossível testar tudo.

Por isso o sistema usa heurísticas — algoritmos que encontram uma resposta muito boa (mas não necessariamente perfeita) em milissegundos.

1 Visão Geral do Pipeline

O sistema executa 4 etapas em sequência ao clicar em Otimizar Rota:

1
Montar a Matriz de Distâncias por Estrada
Uma chamada à API OSRM retorna a distância real de asfalto entre todos os pares de pontos.
2
Nearest Neighbor — construir a rota inicial
Algoritmo ganancioso: sempre vai para o ponto mais próximo ainda não visitado.
3
2-Opt — refinar a rota
Testa trocas de arestas para eliminar cruzamentos e encurtar o percurso total.
4
Desenhar as Estradas Reais
Para cada trecho da rota otimizada, a OSRM retorna o caminho exato de asfalto e o sistema desenha no mapa.

2 Etapa 1 — Matriz de Distâncias por Estrada

Antes de qualquer algoritmo, o sistema precisa saber quanto custa ir de cada ponto para cada outro ponto. Esse custo é a distância real por estrada em metros.

Por que não usar linha reta (Haversine)?

A distância em linha reta ignora rios, montanhas, sentido das ruas, estradas federais e desvios. SP → RJ em linha reta são ~360 km, mas pela estrada são ~430 km. Otimizar com dados errados gera uma rota errada.

Como funciona — OSRM Table API

O sistema envia uma única chamada HTTP para o servidor OSRM (Open Source Routing Machine, que usa dados do OpenStreetMap) passando todos os pontos de uma vez. O servidor retorna uma tabela N×N com a distância e o tempo de viagem entre todos os pares. 

// Exemplo: 3 pontos → 1 chamada → matriz 3×3
GET https://router.project-osrm.org/table/v1/driving/
    -46.63,-23.55;   // São Paulo
    -43.17,-22.90;   // Rio de Janeiro
    -43.93,-19.91    // Belo Horizonte
    ?annotations=distance,duration

// Resposta:
{
  "distances": [
    [0,      431000, 586000],  // de SP  para: SP, RJ, BH
    [431000, 0,      434000],  // de RJ  para: SP, RJ, BH
    [586000, 434000, 0     ]   // de BH  para: SP, RJ, BH
  ],
  "durations": [  // segundos
    [0,     20100, 27300],
    [20100, 0,     20400],
    [27300, 20400, 0    ]
  ]
}

Com essa matriz em mãos, o algoritmo sabe exatamente o custo de qualquer trecho. A partir daqui, trabalha só com números — rápido e sem mais chamadas de rede.

Eficiência

Para N pontos: apenas 1 chamada HTTP para obter N² distâncias. Não importa se são 5 ou 50 pontos — continua sendo 1 chamada.

3 Etapa 2 — Nearest Neighbor (Vizinho Mais Próximo)

Com a matriz pronta, o sistema constrói a primeira versão da rota. O algoritmo Nearest Neighbor é ganancioso: a cada passo, toma a melhor decisão local — sem planejar o futuro.

Regra simples

Lógica do algoritmo

1. Começa no ponto P1.
2. Olha para todos os pontos ainda não visitados.
3. Vai para o mais próximo (menor distância na matriz).
4. Marca esse ponto como visitado e repete o passo 2.
5. Quando não sobrar nenhum ponto, volta para P1 fechando o circuito.

Exemplo visual com 5 pontos

Pontos: A B C D E Começa em: A Não visitados: B C D E Passo 1 — De A, distâncias: B=120, C=340, D=80, E=210 → Escolhe D (80km, o mais próximo) Passo 2 — De D, distâncias: B=150, C=200, E=90 → Escolhe E (90km) Passo 3 — De E, distâncias: B=130, C=300 → Escolhe B (130km) Passo 4 — De B, distâncias: C=180 → Escolhe C (único restante) Fecha circuito: C → A (220km) Rota: A → D → E → B → C → A Total: 80 + 90 + 130 + 180 + 220 = 700 km

O código em JavaScript

function nearestNeighbor(pts, M) {
  const n       = pts.length;
  const visited = new Uint8Array(n);   // 0 = não visitado
  const order   = [], steps = [];

  let cur = 0;          // começa no ponto 0
  visited[0] = 1;
  order.push(0);

  for (let s = 1; s < n; s++) {
    let best = -1, bestDist = Infinity;
    const searched = [];

    // Varre todos os não-visitados
    for (let j = 0; j < n; j++) {
      if (!visited[j]) {
        searched.push(j);
        if (M[cur][j] < bestDist) {    // encontrou um mais próximo?
          bestDist = M[cur][j];
          best = j;
        }
      }
    }

    steps.push({ from: cur, to: best, dist: bestDist, searched });
    visited[best] = 1;
    order.push(best);
    cur = best;
  }

  order.push(0);              // fecha o circuito
  return { order, steps };
}
Qualidade do resultado

O Nearest Neighbor não garante a rota ótima. Como é ganancioso, pode tomar uma decisão boa agora que cria um problema depois. Em média, produz uma rota 20–25% acima do ótimo. É aí que o 2-Opt entra.

4 Etapa 3 — 2-Opt (Refinamento)

O 2-Opt é um algoritmo de melhoria local. Ele pega a rota do Nearest Neighbor e tenta encurtá-la trocando pares de arestas. Continua tentando até não encontrar mais nenhuma melhoria.

A ideia central: desfazer cruzamentos

Rotas com cruzamentos são sempre piores do que rotas sem cruzamentos. O 2-Opt identifica esses cruzamentos e os desfaz invertendo o segmento entre eles.

Antes — rota com cruzamento
  A ────────── C
   \          /
    \        /
     \      /
      \    /
       B──D

A→B→C→D: ineficiente

Depois — sem cruzamento
  A ── B
  |    |
  |    |
  D ── C

A→D→C→B: mais curto

Como funciona o teste de troca

Para cada par de arestas (A→B) e (C→D), o algoritmo testa se seria melhor ter (A→C) e (B→D), invertendo o segmento do meio. A conta é simples:

// Situação atual: Aresta 1: A → B (custo: M[A][B]) Aresta 2: C → D (custo: M[C][D]) Total atual = M[A][B] + M[C][D] // Proposta de troca (inverte segmento B..C): Nova aresta 1: A → C (custo: M[A][C]) Nova aresta 2: B → D (custo: M[B][D]) Total proposto = M[A][C] + M[B][D] // Troca se: if (M[A][C] + M[B][D] < M[A][B] + M[C][D]) { // inverte o segmento entre B e C → MELHORA! improved = true }

O código em JavaScript

function twoOpt(order, M) {
  let route    = order.slice(0, -1);   // remove duplicata do fim
  const n      = route.length;
  let improved = true;

  while (improved) {              // continua até não melhorar mais
    improved = false;
    for (let i = 0; i < n - 1; i++) {
      for (let j = i + 2; j < n; j++) {
        const a = route[i], b = route[i+1];
        const c = route[j], d = route[(j+1) % n];

        if (M[a][c] + M[b][d] < M[a][b] + M[c][d] - 1e-10) {
          // Inverte o segmento entre i+1 e j
          let lo = i+1, hi = j;
          while (lo < hi) {
            [route[lo], route[hi]] = [route[hi], route[lo]];
            lo++; hi--;
          }
          improved = true;
        }
      }
    }
  }

  route.push(route[0]);    // fecha o circuito
  return route;
}
Ganho típico do 2-Opt

O 2-Opt consegue melhorar a rota do Nearest Neighbor em 10–30% na maioria dos casos, eliminando todos os cruzamentos óbvios. O algoritmo para quando nenhum par de trocas produz melhoria — isso é chamado de ótimo local.

5 Etapa 4 — Traçar as Estradas Reais

Até aqui o algoritmo definiu a ordem de visita ideal. Agora o sistema precisa mostrar o caminho real pelo asfalto entre cada par de pontos consecutivos.

OSRM Route API

Para cada trecho da rota otimizada (ex: SP→BH), o sistema faz uma chamada à OSRM Route API que retorna a geometria exata do caminho por estradas — todas as curvas, rotatórias e vias. 

// Para cada par de pontos consecutivos na rota:
GET https://router.project-osrm.org/route/v1/driving/
    -46.63,-23.55;   // SP
    -43.93,-19.91    // BH
    ?overview=full&geometries=geojson

// Resposta:
{
  "routes": [{
    "distance": 586200,     // metros
    "duration": 27180,      // segundos
    "geometry": {
      "coordinates": [       // centenas de pontos do caminho
        [-46.63, -23.55],   // [longitude, latitude]
        [-46.62, -23.52],
        [-46.58, -23.46],
        // ... mais 300 pontos ...
        [-43.93, -19.91]
      ]
    }
  }]
}

O Leaflet.js recebe esses pontos e desenha a linha exata no mapa, seguindo cada curva da estrada.

Otimização de velocidade: Promise.all

Para N trechos, o sistema dispara todas as N chamadas ao mesmo tempo em paralelo com Promise.all. Se forem 20 trechos e cada chamada demora 500ms, o total é ~500ms — não 10 segundos como seria de forma sequencial. 

// Todas as chamadas em paralelo
const fetches = optOrder.slice(0, -1).map((idx, i) =>
  fetchRoadPath(pts[optOrder[i]], pts[optOrder[i+1]])
);

const paths = await Promise.all(fetches);  // espera todas juntas

paths.forEach(path => {
  L.polyline(path, { color: '#818cf8', weight: 4 }).addTo(map);
});

Qualidade da solução

A combinação Nearest Neighbor + 2-Opt encontra rotas em média 5–15% acima do ótimo teórico para instâncias aleatórias. Na prática, com dados de estradas reais, o resultado é muito bom e computacionalmente milhões de vezes mais rápido que a força bruta.
